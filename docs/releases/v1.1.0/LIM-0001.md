# LIM-0001: String switch hashCode pattern

**Type**: Limitation
**Status**: RELEASED
**Severity**: MEDIUM
**Created**: 2026-03-23
**Resolved**: 2026-03-24

## Description
String switch statements were not fully reconstructed in edge cases involving hash collisions. The two-phase compilation pattern (hashCode switch + equals verification) was not always correctly merged back into a single switch statement.

## Technical Analysis
Java compiles `switch (str)` to a two-phase pattern:
1. Phase 1: `switch (str.hashCode())` maps to an integer index via lookupswitch, with `equals()` verification for each case
2. Phase 2: `switch (index)` dispatches to the actual case bodies

The `StringSwitchReconstructor` was correctly detecting this pattern but had issues with the selector variable -- it was using a temporary copy variable instead of the original switch expression.

### Suggested Approach
Use the original variable as the selector instead of the temp copy in the reconstructed string switch.

### Affected Modules
- `src/main/java/it/denzosoft/javadecompiler/service/converter/transform/StringSwitchReconstructor.java`

### Estimated Effort
LOW

## Solution Implemented
Fixed the string switch reconstructor to use the original variable as the selector, not the temp copy (LIM-0001-20260324-1).

### Files Modified
- `src/main/java/it/denzosoft/javadecompiler/service/converter/transform/StringSwitchReconstructor.java` - Use original variable as selector (line ~82)

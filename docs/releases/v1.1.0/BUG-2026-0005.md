# BUG-2026-0005: Try-catch variable scope

**Type: Bug
**Status**: RELEASED
**Severity**: HIGH
**Created**: 2026-03-23
**Resolved**: 2026-03-24

## Description
Exception variables in catch blocks had incorrect scope handling. The exception variable was either pre-declared outside the try-catch, had the wrong name, or the initial exception store statement was not removed, producing redundant assignments at the start of catch blocks.

## Technical Analysis
The Java compiler reuses local variable slots between try blocks and catch handlers. When the Local Variable Table (LVT) has entries for both the try-body variable and the catch exception variable sharing the same slot, the decompiler was:
1. Picking the wrong variable name (try-body name instead of exception name)
2. Not removing the synthetic `astore` at the start of catch handlers
3. Pre-declaring exception variables outside the try-catch scope

### Suggested Approach
Build a map of handler PC to exception variable names from the LVT. Use handler-specific names instead of try-body names for shared slots. Remove the initial exception store in catch handlers. Exclude exception handler variables from pre-declarations.

- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java`

HIGH

## Solution Implemented
Comprehensive fix across two files with 13 change tags:
1. Built handler PC to exception variable name map from LVT (BUG-2026-0005-20260324-6, -7)
2. For shared slots (try/catch), prefer try-body entry for general use but handler-specific name in catch blocks (BUG-2026-0005-20260324-3, -4)
3. Exclude exception handler catch variables from pre-declarations (BUG-2026-0005-20260324-1, -2)
4. Prefer handler-specific var name from LVT in catch reconstruction (BUG-2026-0005-20260324-8)
5. Remove initial exception store by name match or slot reuse (BUG-2026-0005-20260324-9, -10)
6. Rename misnamed exception variable in handler body (BUG-2026-0005-20260324-12)
7. Helper to find slot variable name at handler PC (BUG-2026-0005-20260324-13)
8. Helper to check if a statement is a null-assignment artifact (BUG-2026-0005-20260324-11)

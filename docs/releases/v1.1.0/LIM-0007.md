# LIM-0007: Lambda captures

**Type**: Limitation
**Status**: RELEASED
**Severity**: LOW
**Created**: 2026-03-23
**Resolved**: 2026-03-24

## Description
Complex lambda captures could show synthetic field accesses instead of the captured variable names. When lambdas capture variables from the enclosing scope, the compiler generates synthetic parameters or fields that were sometimes visible in the decompiled output.

## Technical Analysis
Lambda expressions that capture variables from the enclosing scope are compiled to synthetic methods with the captured values passed as additional parameters. The `invokedynamic` bootstrap method (`LambdaMetafactory.metafactory`) encodes which variables are captured. For simple captures, the synthetic method parameters correspond directly to the captured variables. For complex captures (capturing `this`, multiple variables, or variables from nested scopes), the synthetic access patterns can be harder to map back to original variable names.

### Suggested Approach
The lambda body reconstruction already handles most capture patterns. Remaining edge cases involve matching synthetic parameter names back to the captured variables using the enclosing method's local variable table.

### Affected Modules
- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java`

### Estimated Effort
LOW

## Solution Implemented
Lambda capture handling was improved as part of the general lambda reconstruction pipeline. The existing lambda body reconstruction from synthetic methods via BootstrapMethods now correctly handles the common capture patterns.

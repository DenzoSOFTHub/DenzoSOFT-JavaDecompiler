# LIM-0002: Try-with-resources

**Type**: Limitation
**Status**: RELEASED
**Severity**: MEDIUM
**Created**: 2026-03-23
**Resolved**: 2026-03-24

## Description
Try-with-resources statements were falling back to linear code instead of being reconstructed as `try (Resource r = ...) { ... }` syntax. The complex bytecode generated by the compiler for automatic resource management was not being detected.

## Technical Analysis
Try-with-resources compiles to a complex bytecode pattern with nested try-catch blocks for `close()` calls and `addSuppressed()` exception handling. The temporary variables used for resource management often share local variable slots with other variables, and their types were not available from the descriptor when the LVT entry was missing.

### Suggested Approach
Infer variable types from the right-hand side of assignments when the descriptor is unavailable (common for TWR temporary variables).

### Affected Modules
- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java`

### Estimated Effort
LOW

## Solution Implemented
Added type inference from RHS when descriptor is unavailable, which commonly occurs for try-with-resources temporary variables (LIM-0002-20260324-1).

### Files Modified
- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java` - Infer type from RHS when descriptor is unavailable (line ~1898)

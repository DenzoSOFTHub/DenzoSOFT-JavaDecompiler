# BUG-2026-0043: Malformed `import ::Ljava...;` from signature scanner

**Type**: Bug
**Status**: RELEASED
**Severity**: HIGH
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

Files with generic methods whose type-parameter was a single letter followed by an interface-only bound produced a malformed import line, e.g.:

```
import ::Ljava.lang.foreign.MemoryLayout;
```

Observed in `jdk/internal/foreign/Utils.java` among others.

## Technical Analysis

`collectSignatureImports` in `JavaSourceWriter` scanned the raw JVM signature string looking for `L...;` class descriptors. The JVM generic-signature for an interface-only bound uses a *double* colon, e.g.
```
<L::Ljava/lang/foreign/MemoryLayout;>
```
The scanner saw the character `L` at position 1 (the type-parameter *name* "L") and treated it as the start of a class descriptor, reading forward until the next `;`. The captured "internal name" was therefore `::Ljava/lang/foreign/MemoryLayout`, which after `/ → .` translation produced the malformed import.

### Suggested Approach

Rewrite the scanner to be structurally aware: first consume the optional type-parameter declaration block (`<Name:Bound[:Bound]*;...>`) by skipping the name and parsing each bound via a recursive type-scanner that understands `L`, `T`, `[`, primitives, and nested type-argument blocks `<...>`. Plus a defensive validator in `addImport` to reject names containing characters outside `/$_A-Za-z0-9` or starting with a digit or slash.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Estimated Effort

MEDIUM

## Solution Implemented

`collectSignatureImports` fully rewritten as a structural parser with helpers `scanSignatureType` and `scanSignatureTypeArgs`. Added `isValidInternalName` guard in `addImport` as a belt-and-braces defence against future regressions.

### Files Modified

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java` — new scanner + validator.

### Test Case

- Verified on `java.base` (3,372 classes): 0 malformed imports remain (was 6 in v1.7.0).

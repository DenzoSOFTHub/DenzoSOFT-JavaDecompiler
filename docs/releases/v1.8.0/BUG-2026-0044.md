# BUG-2026-0044: Text-block emission triggers illegal-delimiter / illegal-escape compile errors

**Type**: Bug
**Status**: RELEASED
**Severity**: HIGH
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

Any String constant with ≥ 2 newline characters was emitted as a Java text block (`"""..."""`). Some of those strings had content that was not safe for the text-block lexer, producing compile errors such as:

```
illegal text block open delimiter sequence, missing line terminator
illegal escape character
```

Observed in `java/util/Scanner.java` (regex patterns containing `\"""`) and similar files.

## Technical Analysis

Text blocks have lexical constraints beyond "has newlines":

- Cannot contain `"""` (premature close).
- Trailing `"` merges with the closing delimiter.
- `\r` is normalized to `\n` by the compiler — changes the value if preserved raw.
- `\` requires re-escaping.
- Line terminators other than `\n` (U+000D standalone, U+0085, U+2028, U+2029) break the lexer.

The previous code assumed any multi-newline string was a safe text block and inserted the raw content between `"""\n` and `"""`.

### Suggested Approach

Guard text-block emission with an `isTextBlockSafe` predicate that rejects any of the unsafe constructs. Fall back to a normal escaped string literal otherwise. Independently, strengthen `escapeString` to `\u`-escape all control characters and the problematic line terminators.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Estimated Effort

LOW

## Solution Implemented

- Added `isTextBlockSafe(String)` guard.
- `escapeString` now `\u`-escapes any char < 0x20 (except `\n`, `\t`), `0x7f`, and `0x85 / 0x2028 / 0x2029`.

### Files Modified

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Test Case

- Verified on `java.base`: all 3 `illegal text block open delimiter` errors and all `illegal character` / `illegal escape` errors from v1.7.0 are gone.

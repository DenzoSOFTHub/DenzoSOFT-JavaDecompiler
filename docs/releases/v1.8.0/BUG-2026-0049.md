# BUG-2026-0049: Reserved-word class/field names rendered verbatim

**Type**: Bug
**Status**: RELEASED
**Severity**: MEDIUM
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

Inner classes whose name collides with a Java reserved word (e.g., `SystemModules$default`) rendered as:

```java
return new SystemModules.default();
```

`default` is a Java keyword so this is illegal. Observed in `jdk/internal/module/SystemModulesMap.java`.

## Technical Analysis

Name sanitization via `IdentifierSanitizer` was only applied when the `--deobfuscate` CLI flag was set. In normal mode the `sn` helper passed names through unchanged. Reserved-word collisions are, however, an *always-illegal* condition in source code regardless of deobfuscation — the class file format allows the name but the grammar doesn't.

### Suggested Approach

In the `sn` helper, unconditionally prepend `_` when the name is a Java keyword (using `IdentifierSanitizer.isKeyword`). Leave all other name handling untouched to preserve byte-for-byte fidelity when no deobfuscation is requested.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Estimated Effort

LOW

## Solution Implemented

Two-line guard in `sn(name, category)`.

# BUG-2026-0045: Non-printable / line-terminator chars in string literals break compilation

**Type**: Bug
**Status**: RELEASED
**Severity**: MEDIUM
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

String constants containing control characters or Unicode line terminators (U+0085 NEL, U+2028 LINE SEPARATOR, U+2029 PARAGRAPH SEPARATOR, etc.) were emitted verbatim, producing javac errors like `illegal character: '\u2028'` or `illegal character: '\ue800'`.

## Technical Analysis

`escapeString` only had explicit cases for `\n \r \t \b \f \\ \"`. Everything else was appended as-is. JLS treats 0x0A, 0x0D, 0x85, 0x2028, 0x2029 as line terminators even *inside* a string literal, so emitting them raw breaks lexing.

### Suggested Approach

Extend the default branch of the `escapeString` switch to `\u`-escape any char < 0x20 (except already-handled `\n` / `\t`), `0x7f`, and the three extra Unicode line terminators.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Estimated Effort

LOW

## Solution Implemented

Added `isProblematicLineTerminator` helper and extended `escapeString` default branch accordingly.

### Files Modified

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Test Case

- `java.base`: all `illegal character` errors from v1.7.0 eliminated (22 errors across multiple `CharacterData*.java`, `Spliterator.java`, etc.).

# BUG-2026-0051: Multi-value operand stack lost across block boundaries (ternary-in-expression)

**Type**: Bug
**Status**: RELEASED
**Severity**: HIGH
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

Compound expressions containing a ternary sub-expression (very common in Lombok-generated `hashCode` / `equals`) produced spurious STACK_UNDERFLOW markers on the following arithmetic operation, e.g.:

```java
public int hashCode() {
    // === DECOMPILATION NOTES (body may be inaccurate) ===
    //   - STACK_UNDERFLOW pc=27 opcode=0x60 (int placeholder used)
    int PRIME = 59;
    int result = 1;
    Object $idUser = this.getIdUser();
    result = $idUser == null ? 43 : $idUser.hashCode();   // lost `result * PRIME +`
}
```

## Technical Analysis

In bytecode, compound arithmetic around a ternary looks like:

```
iload result
iload PRIME
imul                 ; stack: [result*PRIME]
<condition block>    ; leaves 43 or hashCode() on stack
; target block: stack is now [result*PRIME, ternary_result]
iadd                 ; expects two values
```

`BasicBlock` only tracked `stackTopExpression` — the top-of-stack expression at block exit. When a successor block inherited state from its predecessor, it inherited only the top value, losing everything below. The `iadd` in the target block then saw only one value (`result*PRIME` was gone) and fell back to a placeholder.

### Suggested Approach

Snapshot the *full* exit stack at the end of each block (not just the top). On block entry, restore the snapshot from an appropriate predecessor. Keep the legacy single-slot `stackTopExpression` behaviour as a fallback for blocks without a snapshot.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/converter/cfg/BasicBlock.java`
- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java`

### Estimated Effort

LOW (storage was a List, seeding path was already localized)

## Solution Implemented

- `BasicBlock.exitStack` (List<Expression>, bottom-first) added.
- Decoder snapshots the stack at block exit.
- On entry, if any predecessor has a non-empty `exitStack`, the successor pushes all entries in push-order; else it falls back to the old `stackTopExpression` logic.

### Files Modified

- `src/main/java/it/denzosoft/javadecompiler/service/converter/cfg/BasicBlock.java`
- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java`

### Test Case

- On `contrp.be-springboot-22.2.66.jar`: STACK_UNDERFLOW incidents 864 → **0** after this fix (combined with BUG-2026-0050).
- Trade-off noted: one new failure on `jdk/internal/constant/DirectMethodHandleDescImpl.java` where the richer stack inheritance reaches a pattern the decoder still can't represent. Kept because the overall impact is strongly positive (hundreds of incidents resolved).

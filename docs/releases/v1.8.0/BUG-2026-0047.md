# BUG-2026-0047: Interface static initializer block rendered (illegal in Java interfaces)

**Type**: Bug
**Status**: RELEASED
**Severity**: HIGH
**Created**: 2026-04-20
**Resolved**: 2026-04-20

## Description

Interfaces containing `static final` fields initialized from `<clinit>` were emitted as:

```java
public static interface Handler {
    public static final Signal.Handler SIG_DFL;    // no initializer
    public static final Signal.Handler SIG_IGN;
    static {
        SIG_DFL = new Signal.NativeHandler(0L);
        SIG_IGN = new Signal.NativeHandler(1L);
    }
}
```

Interfaces in Java cannot contain static initializer blocks (JLS §9.2), so this fails compilation with `initializers not allowed in interfaces`. Observed in `jdk/internal/misc/Signal.java`, `java/util/zip/Checksum.java`, `java/lang/classfile/TypeAnnotation.java`.

## Technical Analysis

Two independent shortcomings:

1. The top-level class writer already inlined clinit assignments into field declarations, but the inner-class writer did not — so nested interfaces (Handler is inside Signal) kept clinit as a block.
2. Even after inlining, if clinit contained any residual statement (assertion bootstrap, etc.), the `static { ... }` block was still emitted inside an interface.

### Suggested Approach

- Replicate the "inline clinit assignments into field declarations" logic in the inner-class writer path.
- Unconditionally skip `<clinit>` when the containing type is an interface, at both top level and inner. Any clinit content that cannot be inlined is synthetic and safe to drop.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Estimated Effort

LOW

## Solution Implemented

- Inner-class field writer now consults `innerStaticInits` and uses `writeFieldWithInit` for matches.
- Two new guards in the method loop skip `<clinit>` in interface bodies (top-level and inner).

### Files Modified

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

### Test Case

- Verified on `java.base`: all 3 `initializers not allowed in interfaces` errors eliminated.

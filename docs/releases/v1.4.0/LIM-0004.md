# LIM-0004: Type annotations (Java 8+) parsed but not rendered

**Type**: Limitation
**Status**: OPEN
**Severity**: LOW
**Created**: 2026-03-23

## Description
RuntimeVisibleTypeAnnotations and RuntimeInvisibleTypeAnnotations attributes (JVM spec 4.7.20, 4.7.21) are parsed at the bytecode level -- the attribute bytes are read and skipped in `AttributeParser` -- but the annotation information is not extracted or rendered in the decompiled output.

Type annotations appear on:
- Field types: `@NonNull String name;`
- Method return types: `public @NonNull String getName()`
- Parameter types: `void process(@NonNull String s)`
- Type arguments: `List<@NonNull String>`
- Extends/implements clauses, throws clauses, type casts, instanceof, new expressions

## Technical Analysis
Type annotations (Java 8+, JSR 308) use a complex structure defined in JVM spec 4.7.20:

1. **target_type** (1 byte): Identifies what the annotation targets (field type, return type, parameter type, type argument in generic, etc.) -- 22 possible target types.
2. **target_info** (variable length): Depends on target_type. Includes type_parameter_target, supertype_target, type_parameter_bound_target, empty_target, formal_parameter_target, throws_target, localvar_target, catch_target, offset_target, type_argument_target.
3. **type_path** (variable length): Specifies which part of a compound type the annotation applies to (e.g., which type argument of `Map<@A String, @B Integer>`, or the element type of `@C int[][]`).
4. **Annotation data**: Same format as regular annotations (type descriptor + element-value pairs).

Full rendering requires:
- Parsing all 22 target_type variants with their variable-length target_info
- Interpreting type_path entries to map annotations to specific positions within complex generic types
- Modifying the type rendering pipeline in `JavaSourceWriter` to insert annotations at correct positions within type expressions

### Suggested Approach
For a future version, implement full type_annotation parsing in `AttributeParser.java` and store the annotations on the `JavaSyntaxResult` model. The writer would then need to be enhanced to position annotations correctly within type expressions.

### Affected Modules
- `src/main/java/it/denzosoft/javadecompiler/model/classfile/attribute/AttributeParser.java` - Currently skips bytes at lines 112-117
- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java` - Would need annotation insertion in type rendering

### Estimated Effort
HIGH - Requires parsing 22 target_type variants, type_path resolution, and modifying the type rendering pipeline to position annotations within complex generic type expressions.

## Reason for Remaining Open
Full rendering requires JVM spec 4.7.20 type_path resolution which maps annotations to specific positions within complex generic types. The attribute bytes are correctly consumed (skipped) so class file parsing is not affected. This is a cosmetic limitation -- the decompiled code is functionally correct, just missing type annotations.

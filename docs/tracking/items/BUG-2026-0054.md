# BUG-2026-0054: Try-with-resources emitted without type declaration

**Type**: Bug
**Status**: TO_ANALYZE
**Severity**: MEDIUM
**Created**: 2026-04-21

## Description

Try-with-resources blocks render using an already-declared local variable as the resource head, without emitting the required type declaration:

```java
try (out = new DataOutputStream(new BufferedOutputStream(new FileOutputStream(filename)))) {
    cf.write(out);
} catch (IOException e) {
    throw e;
}
```

`error: the try-with-resources resource must either be a variable declaration or an expression denoting a reference to a final or effectively final variable`

Two legal shapes:
- Java 7–8: `try (DataOutputStream out = new DataOutputStream(...))` — declaration required
- Java 9+: `try (out)` — bare identifier only (must be effectively final)

The emitted `try (out = ...)` is neither.

## Technical Analysis

`TryCatchReconstructor` recognises the try-with-resources pattern from the `close()` invocations in finally, but when the resource variable was declared separately before the try, it emits the assignment-form `out = expr` as if it were a declaration. The writer path for the resource slot needs to prepend the resource type (looked up from LocalVariableTable or the expression return type).

Occurrences observed:
- `javassist/util/proxy/FactoryHelper.java:221`

### Suggested Approach

In the TWR resource emission, always prepend the resource's Java type (from LVT if available, otherwise inferred from the RHS expression type). Alternatively, if Java 9+ and the resource is effectively final, prefer the bare-identifier form.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/converter/transform/TryCatchReconstructor.java`
- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

LOW-MEDIUM

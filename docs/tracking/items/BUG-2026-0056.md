# BUG-2026-0056: Try-catch-finally: catch body truncated and post-try code dropped

**Type**: Bug
**Status**: TO_ANALYZE
**Severity**: HIGH
**Created**: 2026-04-21

## Description

When a method has:

- a try block with a `finally` implemented via BOTH the Java-compiler's pattern of *inlining* the finally on the normal path AND a synthetic any-catch handler;
- a catch block that composes an exception and throws it;
- additional statements *after* the try-catch-finally.

The decompiler truncates the catch body after the first assignment and drops every statement that follows the try block. The method ends without a return for a non-void signature.

Example from `SimogClient.consultaGara(String)` (contrp.be-springboot-22.2.66.jar):

Expected (reconstructed from bytecode):
```java
public ConsultaGaraResponse consultaGara(String cig) throws MyException {
    log.info("consultaGara - start: cig '{}', URL '{}'", cig, getWebServiceTemplate().getDefaultUri());
    String ticket = login(username, password);
    ...
    ConsultaGaraResponse resp = null;
    try {
        resp = (ConsultaGaraResponse) ((JAXBElement) getWebServiceTemplate()
            .marshalSendAndReceive(objectFactory.createConsultaGara(createConsultaGara))).getValue();
    } catch (Exception e) {
        String message = "consultaGara - ";
        if (e instanceof WebServiceTransportException && e.getMessage().contains("Unauthorized [401]")) {
            message = message + Utilities.MSG_ANAC_UTENZA_NON_AUTORIZZATA;
        } else {
            message = message + e.getMessage();
        }
        throw new MyException(message);
    } finally {
        logout(ticket);
    }
    log.info("consultaGara - end");
    return resp;
}
```

Current output (v1.8.0):
```java
public ConsultaGaraResponse consultaGara(String cig) throws MyException {
    String message;
    ...
    ConsultaGaraResponse resp = null;
    try {
        resp = (ConsultaGaraResponse) ((JAXBElement) ...).getValue();
    } catch (Exception e) {
        message = "consultaGara - ";          // <-- catch body stops here
    } finally {
        this.logout(ticket);
    }
    // <-- missing: log.info + return resp
}
```

## Technical Analysis

Bytecode layout:
- `try`  pc 77-103
- `finally (inlined on success path)` pc 103-108 → `logout(ticket); goto 187`
- `catch (Exception e)` pc 111-176 → compose message + `athrow new MyException(message)`
- `any-handler (finally)` pc 177-186 → `logout(ticket); athrow` (rethrow)
- `after try` pc 187-199 → `log.info("- end"); return resp`

Exception table:
```
77-103   Exception   -> 111
77-103   any         -> 177
111-179  any         -> 177
```

`TryCatchReconstructor` recognises the inlined-finally + any-handler pair and strips the inline `logout(ticket)` from the success path. Two things then go wrong:

1. **Catch body truncation**: Only the first assignment of the catch block is captured. The subsequent `instanceof` / message compose / throw statements are dropped. Most likely cause: the reconstructor stops walking catch-block statements at the first statement that is itself a try-internal pattern marker, or the `athrow` at the catch's terminator isn't registered as a statement before block tail processing.
2. **Post-try code dropped**: The code after pc 187 (the goto target of the success path) never makes it into the outer statement list. The reconstructor returns the try-catch-finally as the final statement of the method, losing the remaining basic blocks.

### Suggested Approach

- In `TryCatchReconstructor`, after materialising the try/catch/finally bodies, continue walking the block list from the first block whose `startPc >= endPc-of-any-handler` (or the success-path goto target) and append those blocks to the outer statement list.
- Audit the catch-body extraction to ensure all statements between `handlerPc` and the catch's natural exit (athrow / goto / return / next catch-start) are collected.

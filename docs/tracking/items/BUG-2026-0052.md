# BUG-2026-0052: Cast precedence broken in field/method access chain

**Type**: Bug
**Status**: TO_ANALYZE
**Severity**: HIGH
**Created**: 2026-04-21

## Description

Cast expressions applied to a receiver in a longer member-access chain render without the required outer parentheses, producing illegal source:

```java
(Op02WithProcessedDataAndRefs) op2list.get(0).ssaIdentifiers.mergeWith(initial);
(TypeData.TypeVar) t.usedBy.add(this);
```

The compiler interprets `(T) obj.field.method()` as a cast of the whole `obj.field.method()` result, which is then used as a bare expression statement — `error: not a statement`.

## Technical Analysis

The original source is of the form:

```java
((Op02WithProcessedDataAndRefs) op2list.get(0)).ssaIdentifiers.mergeWith(initial);
((TypeData.TypeVar) t).usedBy.add(this);
```

Bytecode pattern:
```
... load obj ...
checkcast T
getfield field
...
invoke method
```

When the writer emits a `CastExpression` whose parent is a `FieldAccessExpression` or `MethodInvocationExpression` with the cast as receiver, it must wrap the cast in an additional pair of parentheses. Currently the cast is emitted as `(T) obj` and the subsequent `.field` / `.method()` binds only to the cast target, not to the cast itself.

Occurrences observed:
- `org/benf/cfr/reader/bytecode/analysis/opgraph/Op02WithProcessedDataAndRefs.java:2002, 2260`
- `javassist/bytecode/stackmap/TypeData.java:728`

### Suggested Approach

In `JavaSourceWriter.writeExpression` at the `CastExpression` branch, detect when the cast is being emitted as the receiver of a field access or method invocation and emit outer parentheses in that context. The cleanest fix is in the callers (`FieldAccessExpression`, `MethodInvocationExpression` writing paths): if the receiver is a `CastExpression`, wrap the whole receiver in `()` before emitting the dot.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java`

LOW

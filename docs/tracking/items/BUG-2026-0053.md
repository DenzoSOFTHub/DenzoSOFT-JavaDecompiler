# BUG-2026-0053: Ternary expression emitted as bare statement in lambda body

**Type**: Bug
**Status**: TO_ANALYZE
**Severity**: HIGH
**Created**: 2026-04-21

## Description

Lambda bodies produced from synthetic BootstrapMethods include a ternary expression used as a bare statement, which is not valid Java:

```java
forEachInvocation(code, cp, (owner, name, desc, pc, it) -> {
    String scheme;
    "java.net.URL".equals(owner) == 0 || "<init>".equals(name) == 0
        ? "java.net.URI".equals(owner) != 0
            ? "<init>".equals(name) == 0 ? "create".equals(name) != 0 ? 1 : 0 : 1
            : 0
        : 1;      // <-- illegal: ternary used as statement
});
```

```java
minSev.addActionListener(() -> {
    AnalysisEngine.Severity threshold;
    String sel = (String) minSev.getSelectedItem();
    sel == null ? "All" : sel;   // <-- illegal: ternary used as statement
});
```

`error: not a statement`.

## Technical Analysis

The existing ternary-as-statement workaround (from v1.6.0, wraps orphan ternary in variable assignment) does not reach through synthetic lambda method bodies — the transformation likely runs on the outer method body only.

Occurrences observed:
- `it/denzosoft/springbootanalyzer/core/analysis/AnalysisEngine.java:1525`
- `it/denzosoft/springbootanalyzer/ui/panels/SecurityAnalysisPanel.java:129`

### Suggested Approach

Apply the ternary-as-statement transformation recursively into synthetic lambda bodies, or move the transformation to a single place in `JavaSourceWriter` that is invoked right before rendering any statement. Alternative: drop orphan ternary statements entirely — if the bytecode produced a value on the stack that was immediately popped, the equivalent source-level statement is a no-op.

### Affected Modules

- `src/main/java/it/denzosoft/javadecompiler/service/converter/ClassFileToJavaSyntaxConverter.java` (synthetic method body decode)
- `src/main/java/it/denzosoft/javadecompiler/service/writer/JavaSourceWriter.java` (lambda body emission)

### Estimated Effort

LOW

# BUG-2026-0055: `new X(arrayArg)` mid-method misrendered as `super(array)` plus stray array

**Type**: Bug
**Status**: TO_ANALYZE
**Severity**: HIGH
**Created**: 2026-04-21

## Description

A local `new JComboBox<>(new String[]{...})` inside a method is reconstructed as two separate statements — a misplaced `super(...)` call (illegal outside a constructor's first statement) and a bare NewArrayExpression assigned to a variable of the wrong type:

```java
top.add(Box.createHorizontalStrut(20));
super(new String[]{"All", "LOW", "MEDIUM", "HIGH", "CRITICAL"});      // <-- illegal here
JComboBox minSev = new String[]{"All", "LOW", "MEDIUM", "HIGH", "CRITICAL"};   // <-- type mismatch
minSev.addActionListener(() -> { ... });
```

Original source is effectively:

```java
JComboBox minSev = new JComboBox<>(new String[]{"All", "LOW", "MEDIUM", "HIGH", "CRITICAL"});
```

`error: not a statement` (triggered by the downstream lambda mishandling; the super/type-mismatch lines by themselves would also fail to compile).

## Technical Analysis

Bytecode for `new JComboBox<>(array)`:
```
new JComboBox
dup
... build array ...
invokespecial JComboBox.<init>([Ljava/lang/String;)V
astore <local>
```

Expected rendering path:
1. `new JComboBox` pushes a `NewExpression` stub.
2. `dup` duplicates the stub.
3. Array literal is built on top.
4. `invokespecial <init>` pops (receiver = duplicated NewExpression, arg = array). The handler recognises the receiver as a `NewExpression` and attaches the arguments.
5. `astore` consumes the original stub (now with args) and assigns it to the local.

Observed rendering indicates step 4 failed to identify the receiver as a `NewExpression` — so the decompiler fell through to the "super/this constructor call" path, emitting `super(array)`. The surviving original stub and/or the array survived on the stack and got picked up by the astore as the RHS, producing the type-mismatch assignment.

The most plausible cause: in `decodeOpcode` at `INVOKESPECIAL` (`case 0xB7`), the pattern match
```java
if (obj instanceof NewExpression) { ... } else { /* emit super/this */ }
```
is defeated when `obj` was retrieved through the new multi-value exit-stack inheritance (BUG-2026-0051): the re-pushed expression may have been wrapped (e.g. in a checkcast when the predecessor ended mid-type-coerce) and no longer satisfies `instanceof NewExpression`.

Occurrence observed:
- `it/denzosoft/springbootanalyzer/ui/panels/SecurityAnalysisPanel.java:127-128`

### Suggested Approach

- In INVOKESPECIAL handling, also match wrapped shapes (`CastExpression` around a `NewExpression`) before falling back to super/this.
- When the receiver is not a NewExpression AND the target class is not the current class or its superclass, emit `new X(args)` (and pair it with the astore if the pattern matches) rather than `super(args)` — `super` in a non-constructor is unambiguously wrong.
